# Phase 6 설명

## 구현 목표

Phase 6에서는 이전 Phase에서 만든 **모집 시스템(study_group)**을 실제 **예약(reservation)**과 결합하여, 그룹 단위로 스터디룸을 예약할 수 있는 기능을 구현했습니다. 개인 단위 예약(1명)과 그룹 예약(current_members)이 모두 공존하며 조화롭게 수용 인원 검증이 이뤄지도록 확장했습니다.

## 주요 변경 사항

### 1. `reservation` 모델 확장
- `group_id`: 외래키(`ForeignKey("study_group.id")`) 컬럼 추가. NULL 허용(선택) 속성을 지님으로써 개인 예약은 NULL, 그룹 예약은 그룹 ID를 기록하는 하이브리드 예약 테이블로 동작하게 만들었습니다.
- 테이블 삭제 시 예약 유지 문제를 해결하기 위해 `ondelete="SET NULL"` 조건을 추가로 적용했습니다.
- `group` relationship을 추가하여 예약 정보에서 손쉽게 그룹 정보나 인원수(`group.current_members`)를 역참조할 수 있도록 연결했습니다.

### 2. 복합적인 예약 제약/검증 로직 (`reservation_service.py`)

기존에는 단순히 *'같은 시간대에 몇 개의 예약이 있는지(count)'*를 기준으로 수용 인원(`capacity`)을 검증했습니다. 이번 과정에서는 실제 가용 인원을 계산하도록 로직 수준을 끌어올렸습니다.

- **조장 권한 및 그룹 상태 검증**:
  - 그룹 예약을 제출할 수 있는 사람은 해당 그룹의 조장(`leader_id`) 뿐입니다.
  - 그룹은 반드시 인원이 다 차서 `"모집완료"` 상태로 닫혀있어야만 예약이 가능합니다.
  - 예약하려는 방의 수용 인원(`capacity`)보다 그룹 인원(`current_members`)이 많을 시 예약을 원천 차단합니다.

- **가용 인원(Occupancy) 누적 검증**:
  - 겹치는 시간대의 타 예약들을 단순히 건수(count)로 세지 않고 인원수로 합산합니다.
  - 타 예약이 개인 예약(`group_id == None`)이면 1명, 그룹 예약이면 해당 그룹의 `current_members` 명으로 판단해 합산합니다.
  - 기존 합계와 새로 신청한 예약의 인원수(개인 1명 흑은 새로운 그룹 인원)를 더했을 때, 방의 수용 인원(`room.capacity`)을 초과하는지 검증합니다.
